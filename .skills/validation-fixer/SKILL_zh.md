# 验证修复 Skill

## 目标

用最小、安全的修改修复验证失败。

这个 skill 用于修复 typecheck/build/test/lint 错误。不能借修错之名重构无关代码，也不能实现未来阶段功能。

## 必要输入

修复前收集：

1. 失败的具体命令。
2. 完整错误输出。
3. 当前 phase 文件。
4. 当前阶段已修改的文件。
5. 相关 package scripts。
6. 最近是否改过依赖。

## 工作流

1. 确认失败命令。
2. 分类错误：
   - TypeScript 类型错误。
   - 构建配置错误。
   - 测试失败。
   - lint 失败。
   - 依赖缺失。
   - 运行路径错误。
   - 脚本配置错误。
3. 定位最小失败模块。
4. 只查看理解错误所需的文件。
5. 用最小 patch 修复根因。
6. 不重构无关代码。
7. 重新运行失败命令。
8. 通过后报告修复结果。
9. 如果仍失败，总共最多尝试 3 次。
10. 3 次后仍失败，停止并输出失败报告。

## 硬性规则

- 不要删除测试来让验证通过。
- 不要削弱测试断言。
- 不要关闭类型检查。
- 未经明确批准，不要放松编译设置。
- 未经明确批准，不要移除 lint 规则。
- 不要绕过验证脚本。
- 未经批准，不要新增依赖。
- 不要通过改变架构来掩盖错误。
- 修复验证时不要实现未来阶段功能。

## 常见修复策略

优先使用这些小修复：

- 补充缺失 export / import。
- 修正路径别名。
- 修复 TypeScript interface 不匹配。
- 对齐 shared DTO 和实现。
- 修正 monorepo package references。
- 修正 `package.json` 里的 script 名称。
- 仅在当前阶段需要时补充测试配置。
- 尽量用窄类型替代不安全的 `any`。

## 失败报告格式

如果无法解决，报告：

1. 失败命令。
2. 错误摘要。
3. 关键错误片段。
4. 疑似根因。
5. 查看过的文件。
6. 尝试过的修复。
7. 为什么仍未解决。
8. 需要人工决定的问题。

## 成功报告格式

如果已解决，报告：

1. 失败命令。
2. 根因。
3. 修改文件。
4. 应用的修复。
5. 重新运行的验证命令。
6. 最终结果。
