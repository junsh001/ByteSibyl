# 代码审查 Skill

## 目标

在阶段完成前审查实现改动。

审查重点是：是否遵守阶段边界、是否保持架构分层、是否符合安全规则、是否通过验证、文档是否同步。

## 必要输入

审查前检查：

1. 当前阶段文件：`docs/milestones/`。
2. `AGENTS.md`。
3. `docs/ARCHITECTURE.md`。
4. 已变更文件。
5. 验证命令输出。
6. 已更新文档。

如果没有运行验证命令，审查结果应标记为不完整。

## 审查范围

### 1. 阶段范围

检查：

- 是否只实现当前阶段？
- 是否实现了被禁止的未来功能？
- 是否修改了无关模块？
- 是否新增了阶段范围外的依赖？

### 2. 架构边界

检查：

- Web UI 是否保持轻量？
- `agent-core` 是否独立于 Web UI？
- 共享协议是否放在 `packages/shared`？
- workspace 执行是否和模型推理分离？
- 工具是否结构化、类型化？

### 3. 安全和权限

检查：

- 文件写入是否显式？
- 命令执行是否受控？
- 危险命令是否被禁止或需要审批？
- 是否存在路径边界限制？
- 是否避免读取或暴露 secrets / env 文件？

### 4. 验证

检查：

- 是否运行了要求的验证命令？
- 是否通过？
- 如果失败，是否诚实报告？
- 是否删除、削弱或绕过测试？
- 是否未经批准放松编译设置？

### 5. 文档一致性

检查：

- 是否更新设计文档？
- 是否更新教程章节？
- 是否更新博客草稿？
- 文档是否和真实实现一致？
- 未完成能力是否明确标记为限制？

## 审查等级

使用以下等级之一：

- `PASS`：当前阶段可接受。
- `PASS_WITH_NOTES`：可接受，但有少量后续建议。
- `CHANGES_REQUIRED`：必须修改后才能结束阶段。
- `BLOCKED`：缺少必要信息、验证结果或治理文件。

## 输出格式

返回：

1. 审查等级。
2. 总结。
3. 阶段范围发现。
4. 架构发现。
5. 安全发现。
6. 验证发现。
7. 文档发现。
8. 必须修复项。
9. 可选优化项。
